# SearchAudit

SearchAudit on FastAPI / Python -backendin ja SvelteKit + Tailwind CSS -frontendin yhdistelmä sisäisen haun termikandidaattien manuaaliseen arviointiin.
Nykyinen release candidate on `0.1.0-rc2`.
Frontend on replatformoitu SvelteKitiksi, mutta backendin API-sopimukset ja CSV-skeema ovat ennallaan.

## Arkkitehtuuri

- Backend: FastAPI / Python
- Frontend: SvelteKit
- Styling: Tailwind CSS build-pipeline frontendissä
- UI-foundation: shadcn-svelteä varten on nyt paikallisesti scaffoldattu `frontend/src/lib/components/ui/`-hakemisto ja `frontend/components.json`; foundation-komponentit ovat `Button`, `Card`, `Badge`, `Tabs`, `Tooltip`, `Textarea`, `Input`, `Label`, `Table`, `Dropdown Menu`, `Empty` ja `Separator`, ja niitä käytetään aktiivisissa MVP-näkymissä (Target Context, Active Filters, Triage Queue, Focus Mode ja LLM Prompts)
- Node/npm ovat sallittuja frontendin rakentamiseen ja kehitykseen
- Reactia ei käytetä
- Vueta ei käytetä

Tämä on edelleen kevyt, paikallisesti ajettava sovellus. LLM API, scraping ja Optimizely-integraatiot ovat edelleen backlogilla.
Uusi frontend sijaitsee hakemistossa `frontend/`, ja vanha `static/`-UI säilyy rinnalla migration-/fallback-tilanteita varten.

## Asennus

Backend:

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

Jos käytät suoraan `start-backend.cmd`-launcheria, se yrittää asentaa backend-riippuvuudet automaattisesti, jos `requests` puuttuu.

Frontend:

```bash
cd frontend

npm run check

```

Tailwind ei ole enää CDN-riippuvainen, vaan se käännetään frontendin build-pipelineen.

## MVP 1.3 API Audit

- Näkyvät MVP-tabit ovat nyt `Triage Queue`, `Focus Mode`, `LLM Prompts` ja `API Audit`.
- API Audit käyttää vain hyväksyttyjä rivejä, muodostaa auditointipyynnön nykyisestä kohdesivuprofiilista ja kutsuu `POST /api/audit/run` -reittiä.
- API Audit tukee kahta moodia: `backend_mock` ja `live_api`.
- `backend_mock` ei tee ulkoisia Search/Predictive API -kutsuja.
- `live_api` kutsuu backendistä Terveystalon Search/Predictive API -endpointteja.
- Results Table on API Audit -näkymän päätyöalue, ja request preview sekä raw evidence ovat collapsible-teknisiä osioita.
- Review- ja rejected-rivejä ei ajeta oletuksena.
- API Audit ei tee Optimizely-muutoksia eikä LLM-kutsuja.
- Jos backend-auditointipäätepiste ei vastaa, näkymä näyttää unavailable-tilan.
- Live API -virheet näkyvät UI:ssa yksittäisinä `result_status=error`-tuloksina.
- Varsinainen Search/Predictive backend-integraatio pysyy backlogilla vain siltä osin kuin sitä ei vielä ole saatavilla ilman sopimusmuutoksia.

## MVP 1.5 Recommendations / Toimenpidematriisi

- API Audit -näkymä sisältää rule-based Recommendations Matrixin, joka muodostaa ihmisen tarkistettavat optimointiehdotukset auditResults-datasta.
- Recommendations käyttää backendin `POST /api/recommendations/generate` -reittiä ja voidaan viedä paikallisesti `POST /api/recommendations/export` -reitillä tiedostoon `output/recommendations.csv` sekä analysoitavana workbookina `POST /api/recommendations/export-xlsx` -reitillä tiedostoon `recommendations_<target-slug>_<timestamp>.xlsx`.
- Kaikkien suositusten `approval_status` on `review`, eikä mitään viedä Optimizelyyn automaattisesti.
- Recommendations perustuu API Audit -tuloksiin ja kohdesivuprofiiliin, mutta ei käytä LLM API:a, scrapingia tai analytiikkaintegraatioita.
- `terms_for_audit.csv`-export säilyy ennallaan, eikä Recommendations muuta sitä.
- ICE-pisteytys on alustava priorisointi, ei lopullinen päätös.

Toimii ilman integraatioita:
- backend_mock auditResults
- rule-based recommendations
- ICE-pisteytys
- recommendations.csv export
- recommendations.xlsx export

Vaatii lähdedataa/tiedostoja:
- hyväksytyt termit
- auditResults
- target profile

Vaatii API/verkkointegraation:
- live_api Search/Predictive API -ajo

Ei kuulu tähän vaiheeseen:
- Optimizely-muutosten vienti
- LLM API
- scraping
- analytiikkaintegraatiot
- automaattinen hyväksyntä

## MVP 1.3 QA ja backend-validointi

- Testit ajetaan komennolla:

```bash
python -m pytest -q
```

- Backend validoi ennen CSV-vientiä, että approved-riveillä on vähintään term, intent_bucket, expected_result, risk ja recommended_test.
- Backend trimmaa syötteet ja estää tyhjien termien viennin.
- Sovellus toimii edelleen ilman integraatioita.
- LLM, scraping ja Search/Predictive API -kutsut ovat edelleen backlogilla.

## MVP 1.5 kohdesivuprofiili ja ChatGPT-prompti

1. Täytä kohdesivuprofiili: `target_url`, `target_name`, `page_type`, `primary_conversion`, `page_summary` ja `exclusions`.
2. Muodosta ChatGPT-prompti sovelluksen painikkeella.
3. Kopioi prompti ChatGPT:hen.
4. Liitä ChatGPT:n CSV-vastaus sovellukseen.
5. Arvioi termit taulukossa.
6. Vie lopuksi vain `approved`-rivit CSV-tiedostoon.

- Sovellus toimii edelleen ilman API-avaimia, LLM-integraatioita, scrapingia tai Search/Predictive API -kutsuja.
- Luonnos talletetaan selaimen localStorageen, joten profiili ja termirivit palautuvat sivun päivityksen jälkeen.

## MVP 1.6 auditointipaketti ja review-työnkulku

1. Täytä kohdesivuprofiili, lisää termit ja arvioi rivit review-työnkulussa.
2. Käytä taulukon filttereitä review_status-, search_category- ja risk-näkymiin.
3. Vie joko vain hyväksytyt rivit tai koko auditointipaketti.

Auditointipaketti kirjoittaa `output`-hakemistoon:

- `output/page_profile.json`
- `output/term_candidates_review.csv`
- `output/terms_for_audit.csv` vain silloin, kun validia approved-joukkoa on

Pakettivienti kirjoittaa myös Excel-katselua varten rinnakkaiset TSV-kopiot:

- `output/term_candidates_review_excel.tsv`
- `output/terms_for_audit_excel.tsv` vain silloin, kun validia approved-joukkoa on
- `output/recommendations.csv` ja `recommendations_<target-slug>_<timestamp>.xlsx` Recommendations Matrixille

- `.csv`-tiedostot ovat pipeline- ja sovelluskäyttöön.
- `.tsv`-tiedostot ovat Excel-katselua varten.
- `.xlsx` on analysoitava workbook, jossa on `Summary`, `Recommendations`, `Evidence` ja `Legend` -sheetit.
- Suomalaisessa Excelissä kannattaa avata `*_excel.tsv`, koska se jakaa sarakkeet oikein.
- Älä tulkitse comma-CSV:n avautumista yhteen soluun datavirheeksi.

Jos validia approved-joukkoa ei ole, `terms_for_audit.csv` jätetään ennalleen eikä sitä ylikirjoiteta tyhjäksi.

- Sovellus toimii edelleen ilman integraatioita, API-avaimia tai erillistä backend-sessiota.
- LLM, scraping, Search/Predictive API -kutsut, Optimizely ja raportointipipeline ovat edelleen backlogilla.

# AI Audit Lab

AI Audit Lab is a local-first AI visibility audit application for systematic review of website content and AI interpretation workflows.

## Repository Context Files

- `AGENTS.md` contains repository-level instructions for AI coding and design assistants.
- `ABOUTME.md` contains broader user/work context for the Local Workflow Workbench.

Current version: `v0.3.3`

## Stack

- Python
- FastAPI
- SQLite
- SQLModel
- Pydantic
- Jinja2
- openpyxl
- Pico.css as a local static base stylesheet
- Minimal CSS
- Uvicorn

## Setup

Use Command Prompt from the project directory:

```cmd
cd "C:\Users\k051602\AIAuditLab"
py -m venv .venv
.venv\Scripts\activate.bat
py -m pip install --upgrade pip
py -m pip install -e .[dev]
```

## Run

```cmd
cd "C:\Users\k051602\AIAuditLab"
run_local.bat
```

The app runs at:

```text
http://127.0.0.1:8000
```

Health endpoint:

```text
http://127.0.0.1:8000/health
```

Pages:

```text
http://127.0.0.1:8000/pages
```

## Page Import

Version `v0.3.3` imports pages from a fixed local file path.

Preferred Excel path:


Templates can declare required and optional variables in front matter. Missing required values are shown as warnings on the render page.

Missing required page context values can be filled as render-time overrides. These override values are used in the prompt preview and saved workflow run snapshot, but they are not saved back to the source Page record.

Rendered prompts remain preview-only until explicitly saved as local workflow runs.

## Workflow Run History

Rendered prompts can be saved as local workflow runs.

AI Audit Lab saves runs with:

```text
workflow_key = "ai_audit"
```

Saved runs store an input snapshot and a rendered prompt artifact. They are local SQLite records and do not call any AI API.

The saved input snapshot includes the ontology-aware render context used at save time, including page identity, template key, page type, entity class, page role description, site section, migration status, journey stage, search intent, and ontology version.

AI responses can be pasted manually into a saved workflow run. Responses are stored locally as `WorkflowArtifact` records with `artifact_type = "ai_response"`. Multiple AI responses can be attached to one run for comparison across providers or models.

Manual audit findings can be added under a specific AI response. Findings are stored locally as `WorkflowArtifact` records with `artifact_type = "audit_finding"`. Findings are manual review notes, not approved medical facts. Use `review_needed = "medical"` for clinical or safety-sensitive issues that require medical review.

Saved findings can be edited, deleted, or reassigned to another AI response in the same workflow run. Reassigning or deleting a finding updates the local response comparison counts.

Manual response scorecards can be added or edited under a specific AI response. Scorecards are stored locally as `WorkflowArtifact` records with `artifact_type = "audit_scorecard"`. They use a 0-3 manual rubric and are not automatic scoring. The clinical safety score is a review aid, not medical approval.

There is no AI API integration. The app does not call ChatGPT, Claude, Gemini, Perplexity, or any external service.

## Projektin tarkoitus

Sivurakenne Workbench on paikallinen päätöstyötila Terveystalon henkilöasiakkaan palvelurakenteen suunnitteluun.

- näyttää päätöstyötä tukevia read-only- ja dry-run-näkymiä
- erottaa lähdearvot, johdetut arvot, tarkistusliput, suositukset ja käyttäjän päätökset toisistaan
- tukee Palvelumaster- ja sivurakennepäätösten valmistelua ilman hiljaista automaattista päättelyä
- auttaa tunnistamaan tarkistettavat aiheet ennen mahdollista tiedon tuontia

Erityinen sisältösääntö:

- Oire- ja sairaus-sivuilla ei ole ajanvarausohjausta
- niillä tulisi olla `Hoitoa tarjoavat asiantuntijat` -lohko

# gsc-cli-lite

Kevyt komentorivityökalu Google Search Console -datan hakemiseen CSV-muodossa.

Paketti sisältää vain olennaiset asiat:

- `gsc_fetch.py` – pieni GSC Search Analytics API -exportteri
- `config/gsc.ini` – valmis asetus GSC-propertylle `sc-domain:terveystalo.com`
- `secrets/gsc-service-account.json` – palvelutiliavain mukana pyynnöstäsi
- `requirements.txt` – tarvittavat Python-riippuvuudet
- `gsc.sh` – macOS/Linux-käynnistin
- `gsc.cmd` – Windows-käynnistin

## Käyttöönotto macOS:llä

Avaa Terminal tässä kansiossa ja aja:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
```

Tarkista ensin, että palvelutilillä on oikeus Search Console -propertyyn:

```bash
./gsc.sh --check-auth
```

Hae viimeiset 28 GSC-päivää sivu + hakulause -tasolla:

```bash
./gsc.sh --preset last28 --dimensions page,query
```

Hae viime kuukauden `/fi/palvelut/`-sivut sivutasolla:

```bash
./gsc.sh --preset last-month --dimensions page --filter page contains /fi/palvelut/
```

Hae tietty kalenterikuukausi hakulauseittain:

```bash
./gsc.sh --month 2026-05 --dimensions query
```

Nopea testi vain 1000 rivillä:

```bash
./gsc.sh --preset last28 --dimensions page,query --max-rows 1000
```

CSV kirjoitetaan oletuksena `output/`-kansioon.

## Käyttöönotto Windowsilla

```cmd
python -m venv .venv
.venv\Scripts\activate
python -m pip install -r requirements.txt
gsc.cmd --check-auth
gsc.cmd --preset last28 --dimensions page,query
```

## Tavalliset argumentit

- `--preset last7|last28|last90|last-month|mtd|ytd`
- `--month YYYY-MM`
- `--start YYYY-MM-DD --end YYYY-MM-DD`
- `--dimensions page,query,date,device,country,searchAppearance`
- `--filter DIMENSION OPERATOR EXPRESSION`
- `--output path/to/file.csv`
- `--max-rows 1000`
- `--dry-run`

Esimerkkejä rajauksista:

```bash
./gsc.sh --preset last28 --dimensions page,query --filter query contains gynekologi
./gsc.sh --preset last28 --dimensions page --filter page includingRegex '/fi/palvelut/.+'
./gsc.sh --month 2026-05 --dimensions date,page --filter device equals MOBILE
```

# Piwik PRO Master Automation Workspace

Local-first application for operational Piwik PRO funnel automation.

Architecture:

- Backend: FastAPI
- Database: SQLite in `~/.piwik-workspace/app.db`
- ORM: SQLModel
- Frontend: SvelteKit SPA
- Styling: Tailwind CSS
- UI approach: shadcn-svelte inspired components
- Deployment: SvelteKit builds static files into `backend/static/`; FastAPI serves API + SPA

## What this workspace does

This starter is designed for the Piwik PRO optimization workflow discussed in the previous sessions:

```text
Piwik PRO API → funnel runs → drop-off calculations → diagnosis cards → priority table → HTML/CSV/Markdown outputs
```

It includes:

- local SQLite persistence
- configurable specialties
- mock demo data
- API client skeleton for Piwik PRO Analytics Query API
- funnel run storage
- priority scoring
- diagnosis card generation
- frontend dashboard for runs, cards and priorities

## Run locally

```bash
chmod +x build_and_run.sh
./build_and_run.sh
```

Open:

```text
http://127.0.0.1:8000
```

API status:

```text
http://127.0.0.1:8000/api/status
```

## First workflow

1. Start the app.
2. Open the dashboard.
3. Click `Run demo automation`.
4. Review priority table and diagnosis cards.
5. Replace mock data with a Piwik PRO API query definition.

## Piwik PRO API integration

1. Open the relevant Piwik PRO report.
2. Use `View API call definition`.
3. Save the query body in `backend/app/automation/piwik_query_blueprint.json` or paste it via your own integration.
4. Set environment variables:

```bash
PIWIK_BASE_URL=https://your-account.piwik.pro
PIWIK_CLIENT_ID=...
PIWIK_CLIENT_SECRET=...
PIWIK_WEBSITE_ID=...
```

## Important interpretation note

If the current report uses `Site searches in session > 0`, the analysis only proves that the session contained internal search. It does not prove the exact keyword or clicked search result. For exact search-result funneling, add an event like:

```text
category = internal_search
action = result_click
name = clicked_result_title
value = result_position
```

with event-scope dimensions:

```text
search_keyword
clicked_result_url
result_position
result_type
specialty
search_results_count
```

- `/` — language selector

# _tools

Local Windows CLI tool. Personal/work command launcher for recurring scripts.

Goal: one command family (`tools <subcommand>` or `t <subcommand>`) instead of remembering folders, filenames, and manual setup steps for each script.

This workspace is local-only. It is intentionally not OneDrive-synced.

## How it works

The runtime launcher lives at `C:\Users\k051602\tools.cmd`, with a short alias at `C:\Users\k051602\t.cmd`. It is intended to be invoked from the user's home folder.

`tools.cmd` is a monolithic Windows `.cmd` dispatcher: one file, one `if /I` branch per subcommand. Each branch invokes a Python script under `scripts\<category>\` using the dedicated venv at `_tools\venvs\default\`.

Source-of-truth copies of the launchers are versioned in `launchers\` and copied to the home folder manually after changes. See `launchers\README.md`.

## Available commands

Current as of the last sync. Run `tools help` for the live list.

- `tools help` — show available commands.
- `tools resize-image INPUT --width WIDTH --height HEIGHT [args]` — resize one image using crop, pad, stretch, or proportional mode.
- `tools url-md URL` — copy one URL's Markdown to the clipboard. Use `--out` or `--output-dir` for optional file export. Terveystalo utility sections are cleaned by default; pass `--include-locations` to keep the location section.
- `tools pages [args]` — browse or search the local palvelusivut CSV by Erikoisala without fetching pages.
- `tools gsc-report --help` / `tools gsc-report --pipeline-check` / `tools gsc-report --dependency-check` — safe CLI-only GSC pipeline entry points owned by `_tools`. They do not launch apps, call live APIs, print credential contents, install dependencies, or write reports.
- `tools gsc-report weekly --help` / `tools gsc-report weekly` / `tools gsc-report weekly --dry-run` — run the local `_tools` weekly skeleton and print a dry-run execution plan only.
- `tools gsc-report weekly --check-auth` — the first live API boundary in `_tools`. It verifies only service-account credential loading plus requested GSC property access. It does not fetch Search Analytics data and does not write files.
- `tools gsc-report weekly --fetch-preview` — the first Search Analytics query boundary in `_tools`. It compares page-level current vs previous rolling windows and prints a compact preview only. It writes no files.
- `tools gsc-report monthly --help` / `tools gsc-report dashboard --help` — local help-only placeholders for future non-UI GSC surfaces.

Weekly contract examples:

```cmd
tools gsc-report weekly --dry-run
tools gsc-report weekly --fetch-preview
tools gsc-report weekly --fetch-preview --window 7
tools gsc-report weekly --site-url sc-domain:example.com --output-dir C:\path\gsc-output --dry-run
tools gsc-report weekly --credentials C:\path\service-account.json --site-url sc-domain:example.com --output-dir C:\path\gsc-output --dry-run
tools gsc-report weekly --check-auth --credentials C:\path\service-account.json --site-url sc-domain:example.com
tools gsc-report weekly --execute --html --site-url sc-domain:example.com --output-dir C:\path\gsc-output
```

`--html` is an optional add-on to `--execute`. When supplied, the weekly run writes a branded, self-contained Finnish HTML report alongside the `.xlsx` workbook in `--output-dir`, using the same filename convention (`.html` instead of `.xlsx`). It reads the in-memory analysis the CLI already builds — no extra API calls, no xlsx round-trip. The HTML report is rendered with the stdlib `string.Template` (no new dependency) from `scripts\gsc\html_report.py` and `scripts\gsc\templates\weekly_report.html`. Brand tokens and colours follow the official Graafinen ohjeisto 01/2026. The real licensed Terveystalo fonts (Terveystalo Sans Display Super for headlines; Terveystalo Sans Regular/SemiBold/Bold for body) live in `scripts\\gsc\\fonts\\` and are base64-embedded into each report at render time, so the output is fully self-contained (no CDN, no external font files) and renders offline. Each report carries roughly 115 KB of embedded fonts.

Active GSC CLI ownership now lives in `_tools` at `scripts\gsc\gsc_cli.py`. `control-panel` is abandoned and may be consulted only as design reference outside the active runtime path.
The real local config path is `config\gsc-report.ini`, which stays ignored by git. The committed example contract is `config\gsc-report.example.ini`.
Credentials are user-managed. The CLI does not copy, move, rewrite, or print credential contents. In this migration step it reports only safe status such as configured/not configured and file exists yes/no.
Future weekly `--execute` is not implemented yet. The weekly skeleton is still pre-workbook. `--check-auth` is limited to credential/property access only, and `--fetch-preview` is limited to page-level Search Analytics preview output only.
Static/semi-manual GSC reporting templates are versioned under `reporting\gsc-templates`. They were imported from the Claude Design prototype ZIP and are not connected to `tools gsc-report` yet. HTML template validation with one real monthly dataset is now the next reporting-design step; workbook output is paused/deferred while that validation happens.
Approved future GSC/SEO/reporting dependencies are recorded for later scoped tasks: `altair`, `pandas`, `advertools`, `openpyxl`, `xlsxwriter`, `requests`, `pyyaml`, `beautifulsoup4`, `markdownify`, `trafilatura`, `google-api-python-client`, `google-auth`, `prophet`, `holidays`, `python-docx`, and `pytest`.

2. Load defaults from `config\gsc-report.ini` and let explicit CLI flags override them.
3. Validate the explicit `--credentials` path and future execution inputs.

5. Use `weekly --fetch-preview` to compare current and previous rolling windows by page without writing files.
6. Validate the imported HTML template bundle manually with one real monthly dataset.
7. Later, transform fetched rows into the chosen report shape and write output only in execute mode, if that remains the chosen behavior.

Current stop point: the implemented surface stops before any workbook generation or report write.

- `tools clean-downloads [args]` — clean the Downloads folder. Dry-run by default; pass `--execute` to actually move files.

## Adding a new command

1. Write the Python script under `scripts\<category>\`. Use stdlib `argparse`. If the script modifies files, default to dry-run and require an explicit `--execute` flag.
2. Add a new `if /I` branch to `launchers\tools.cmd` that invokes the script with the venv Python and any launcher-injected arguments.
3. Copy the updated launcher to the home folder: `copy /Y launchers\tools.cmd C:\Users\k051602\tools.cmd`.
4. Verify with `tools help` and `tools <newcommand> --help`.

# CMS-lohkojen käyttöohje

Staattinen HTML/CSS/vanilla JS -versio Terveystalon CMS-lohkojen käyttöohjeesta. Projektin tarkoitus on tarjota yksi paikallisesti avattava ohjesivu, jossa käyttäjä voi vaihtaa neljän pääosion välillä ilman buildia tai ulkoisia riippuvuuksia.

## Paikallinen käyttö

1. Avaa `index.html` selaimessa.
2. Sivu toimii paikallisesti ilman buildia, pakettien asennusta tai internet-yhteyttä.
3. Julkaistavan näkymän tarvitsemat tiedostot ovat `index.html`, `styles.css`, `script.js` ja mahdollinen `CMS-lohkojen-kayttoohje-assets/`-kansio.

- `index.html`
- `styles.css`

- `TODO-review.md`, jos halutaan näyttää vielä tarkistettavat asiat myös käyttäjäpaketissa

# CV Evidence Template System

A local-first project for generating hyperspecific but safe HTML CVs from:

- a stable candidate profile YAML file
- a reusable impact story library
- a job advertisement
- a Jinja HTML template

The key design decision is separation:

```text
candidate_profile.yml = stable positioning, skills, tools and guardrails
impact_stories.yml = detailed proof points, metrics and STAR stories
job_ad.txt = target role
story_usage_rules.yml = safety, visibility and overclaiming controls
cv_content.json = generated CV content
cv.html = rendered output
```

## Quick start, Windows Command Prompt

```bat
py -m venv .venv
.venv\Scripts\python -m pip install -r requirements.txt
.venv\Scripts\python scripts\validate_data.py
.venv\Scripts\python scripts\run_demo.py
```

Open:

```text
output\demo\cv.html
```

## Quick start, macOS / Linux

```bash
python3 -m venv .venv
./.venv/bin/python -m pip install -r requirements.txt
./.venv/bin/python scripts/validate_data.py
./.venv/bin/python scripts/run_demo.py
```

Open:

```text
output/demo/cv.html
```

## Build a CV for another job ad

Put the job ad text into a `.txt` file, for example:

```text
job_ads/my_target_role.txt
```

Then run:

```bat
.venv\Scripts\python scripts\build_cv.py --job-ad job_ads\my_target_role.txt --out-dir output\my_target_role
```

or on macOS / Linux:

```bash
./.venv/bin/python scripts/build_cv.py --job-ad job_ads/my_target_role.txt --out-dir output/my_target_role
```

## Optional PDF export

Install optional dependencies:

```bat
.venv\Scripts\python -m pip install -r requirements-pdf.txt
```

Then:

```bat
.venv\Scripts\python scripts\build_cv.py --job-ad job_ads\epassi_growth_marketer_b2c.txt --out-dir output\demo --pdf
```

## Privacy warning

The `data/private/` files include real personal CV source data. Do not publish that folder to a public repository.

Use the sanitized examples in `data/examples/` when sharing the project structure.
